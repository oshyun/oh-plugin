# 개발 습관 & 코딩 패턴

내가 개발할 때 일반적으로 따르는 원칙·습관 모음.
프로젝트와 무관하게 적용한다. AI 통합 프로젝트면 C도 적용.

---

## A. 작업 전 사고방식 (핵심 철학)

코드를 쓰기 전에:

- **가정하지 말 것.** 가정을 명시하고, 불확실하면 묻는다.
- **해석이 여러 갈래면 펼쳐 보인다.** 조용히 하나를 고르지 않는다.
- **더 단순한 길이 있으면 말한다.** 정당하면 푸시백한다.
- **헷갈리면 멈춘다.** 무엇이 불명확한지 짚고 묻는다.

판단 기준: "시니어 엔지니어가 이거 과하다고 할까?" → 그렇다면 단순화.

성공 기준을 **검증 가능한 목표**로 바꾼 뒤 그 기준이 충족될 때까지 루프한다.

- "검증 추가" → "잘못된 입력 테스트를 쓰고 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고 통과시킨다"
- "X 리팩토링" → "전후로 테스트가 통과하는지 보장한다"

## B. 코딩 패턴 / 코드 품질 규칙

### 1. 단순함 우선, 외과적 변경
- 요청된 것만. 투기적 기능·추상화·유연성·불가능 시나리오 방어 금지.
- 200줄을 50줄로 줄일 수 있으면 다시 쓴다.
- 기존 코드 편집 시: 인접 코드/주석/포맷을 "개선"하지 않는다.
- 안 망가진 걸 리팩토링하지 않는다. 기존 스타일에 맞춘다.
- 내 변경이 만든 orphan(미사용 import/변수/함수)만 정리. 기존 dead code는 건드리지 않고 **언급만** 한다.
- 테스트: 바뀐 모든 줄이 요청에 직접 추적돼야 한다.

### 2. 불필요한 마이그레이션 코드 금지
- 데이터 파일을 직접 교체할 수 있으면 직접 바꾼다.
- 로더에 normalize/lazy-migration 분기를 끼워넣지 않는다 (한 번 쓰이고 영원히 남는 레거시가 됨).
- 저장 쓰기가 막히면 마이그레이션을 짜는 대신 **권한을 먼저 요청**한다. 코드는 새 스키마만 알게 유지.

### 3. 단일 출처(SSOT)
- 같은 속성의 사본을 여러 곳에 저장하지 않는다.
- 한 곳에서 조회하는 getter를 두고 하위는 그 출처를 신뢰한다.
- 규칙·설정도 단일 출처에 codify하고, 다른 문서는 *왜/사례* 보존용으로만 둔다.

### 4. UI 사전 검증 필수
- 서버에 validation이 있어도 UI에 같은 사전 검증을 넣어 입력 즉시 피드백한다 (에러 메시지 + submit 비활성).
- 빌드 없는 환경이면 로직을 복제하되 양쪽 파일 상단에 `MUST STAY IN SYNC` 주석.
- 이미 로드된 클라 state로 중복·존재 체크를 수행해 서버 왕복을 없앤다.
- 서버 에러 코드는 사용자 언어 메시지로 매핑해 우회·레이스 fallback.

### 5. 헬퍼는 객체 단일 인자, 사소한 래퍼 금지
- `a || b || fallback` 같은 사소한 표현식을 named 함수로 감싸지 않는다 (단일 사용 추상화 금지).
- 헬퍼가 정당하면(여러 호출처가 같은 객체를 다룸) 위치 인자 대신 **도메인 객체 하나**를 받아 내부에서 추출한다.
  - 예: `pickReplyText(result)` (O) vs `pickReplyText(a, b, c)` (X).

### 6. 약어·줄임말 금지
- 새 식별자(변수·함수)와 주석에 의미 불명확한 약어 금지. **풀어쓴 이름** 사용 (`fb` → `fileBrowser`).
- 기존 약어를 일괄 개명할 필요는 없으나, 새로 쓰는 코드·주석은 약어 없이.

### 7. 레거시 식별자 잔재 금지
- 기능 개편 시 옛 설계의 슬롯 번호·모드명이 이름에 남으면(예: `custom1`) 새 설계 기준으로 함께 개명한다.
- 사용자 저장값(localStorage 키 등)이 걸려 있어도 중요도 낮으면 마이그레이션 없이 키를 바꾼다(규칙 2와 결합).

### 8. 확장 포인트엔 관리 주석
- 화이트리스트·레지스트리 등 확장 목록의 정의부 주석에 (1) 추가 절차, (2) 플래그/데이터 흐름, (3) 엣지 케이스를 남긴다.

### 9. 부수 기능에 강한 의존성 금지
- 표시·부가 기능 때문에 런타임 의존성(예: 컨테이너 `.git` 마운트, 런타임 git 호출)을 추가하지 않는다.
- 호스트 스크립트·CI 스텝·기존 데이터 경로 재활용을 우선한다.
- "고작 이 효용에 비해 무거운 수단 아닌가"를 자문한다.

### 10. 중요도 대비 복잡도로 기각 가능
- 리뷰/simplify findings가 추가하는 복잡도가 기능 중요도보다 크면 **한 줄 사유와 함께 기각**한다.
- 오타급·한두 줄 수정은 그대로 적용한다.

### 11. 코드 구현을 사용자에게 넘기지 않는다
- `TODO(human)`·"직접 구현해보세요" 형태로 미루지 않고 모든 코드를 직접 완성한다.

## C. AI 모델·엔진 출력 보정 전략 (QUIRK 태그)

> AI 모델/엔진/게이트웨이를 다루는 프로젝트에만 해당. 일반 웹/CLI 프로젝트면 생략.

AI가 표준대로 동작했다면 없었을 **비표준 출력을 감지·보정·우회하는 코드** 전략.
이런 워크어라운드는 본질적으로 여러 계층에 흩어져 한 파일로 모을 수 없다. 그래서 **코드가 아니라 태그로 인덱싱**한다.

- **태그를 단다.** 포맷: `// QUIRK(<주체>): <어떤 비정상 행동을 보정하는가>`
  - 바로 다음 줄에 제거 조건: `// QUIRK-REMOVE-WHEN: <이 조건이 사라지면 이 코드도 제거>`
  - `<주체>` = 무엇이 잘못 동작하는가: `model`(LLM 출력), `engine`(하네스), 필요시 `litellm` 등.
- **전체 목록은 grep으로 본다.** `grep -rn "QUIRK(" src/` 한 번에 흩어진 우회 지점이 한 목록으로.
- **무엇에 다는가.** 모델·엔진 비정상 행동 보정에만. 일반 방어 코드(입력 검증·권한 체크)에는 달지 않는다.
- **근본 원인을 먼저 본다.** 우리 쪽 원인(시스템 프롬프트·설정)이 자초한 건 아닌지 확인하고, 자초한 quirk는 출처를 함께 고친다.
- **정상화되면 제거한다.** REMOVE-WHEN 충족 시 코드와 태그를 함께 지운다. quirk 우회는 영구 코드가 아니라 *추적되는 임시물*.

## D. 문서화 습관

- **문서는 줄을 짧게.** 한 불릿·문장에 몰아넣지 말고 짧게 끊어 별도 줄로 분리한다.
  - 성격이 다른 내용(동기 vs 동작)은 줄을 나눈다.
- **코드 변경 시 사용자 가이드 동기화 필수.** 기능·동작·UI·플로우가 바뀌면 사용자 가이드/문서도 **같은 작업에서** 갱신한다.
  - API 문서 갱신과 동급으로 취급. 문서 상단의 업데이트 날짜도 함께 갱신.
- **API 변경 시 API 문서 동기화.** 엔드포인트 추가/수정/삭제 시 API 문서를 함께 업데이트한다.

## E. 작업 / 워크플로우 습관 (Git · 커밋)

### E-0. git 환경이면 worktree 기반으로 작업한다 (핵심)

git이 있는 모든 환경에서 **편집은 worktree에서** 한다. main tree는 읽기·조사·머지 전용.
이유: 편집 중 dirty 상태가 다른 작업/머지를 막고, 작업이 격리돼야 깨끗한 브랜치 경계가 남는다.

절차:

1. **첫 편집 전에 worktree부터.** 반드시 `git fetch origin`으로 base를 최신화한 뒤 만든다.
   - `git fetch origin && git worktree add --no-track -b work/<주제> <경로> origin/master`
2. **`--no-track` 필수.** upstream이 `origin/master`로 잡히면 GUI의 pull/sync가 작업 커밋을 기본 브랜치로 직행 push할 수 있다(실제 사고 사례).
3. **모든 Read/Edit는 worktree 경로에서.** 커밋은 `git add <개별 파일>`로 **내 파일만** — `git add -A`·`git add src/` 금지(다른 미커밋 작업이 휩쓸려 들어감).
4. **rebase까지만 worktree에서.** 머지 단계(`checkout master`·`merge`·`push`·`worktree remove`)는 **main tree에서** 실행한다.
   - worktree 안에서 `git checkout master`를 하면 `'master' is already used by worktree`로 실패하고, 체인이 끊기면 작업 브랜치가 원격에 잘못 push된다.
5. **작업 디렉토리가 shallow면** 머지가 unrelated histories로 막힐 수 있다. `git fetch --unshallow origin`으로 복구.

### E-1. 커밋 · 머지 · push

- **작업 완료 후 자동 커밋 + push.** 논리적 단위가 끝나면 묻지 않고 커밋하고 push까지 한다.
- **연속 리팩토링은 push를 마지막에 한 번.** 여러 독립 커밋이 쌓이는 작업은 중간 push 생략, 마지막에 한 번. (CI 중복 빌드 방지)
  - 독립 단일 작업은 완료 즉시 push.
- **커밋 메시지에 작성자 정보(`Co-Authored-By`) 미포함.** 기본값을 두지 말고 작업 내용을 한 줄로 적는다.
- **기본 브랜치에 직접 커밋하지 않는다.** 작업은 작업 브랜치(`work/<주제>`)에서.
- **준선형(semi-linear) 머지.** `fetch → rebase origin/master`로 ff 가능 상태를 만든 뒤, 일부러 `--no-ff` 머지로 작업 경계를 남긴다.
  - 머지 직전 `git fetch`로 base 이후 다른 push(다른 세션 머지·CI bump 등)가 끼었는지 확인하고, 가라앉으면 그 위로 rebase 후 진행.
  - push가 거부되면(race) fetch→rebase→merge를 멈추지 말고 재시도해 원자적으로 반영한다. 충돌 때만 멈추고 알린다.
- **코드 변경 작업을 마치면 simplify/cleanup 패스를 자동 실행** (조사성·질의성 작업엔 미적용).

## F. 에이전트 상호작용 / 응답 스타일 선호

- **자율(오토) 모드 전 예상 소요 시간 제시.** 자율 실행 직전 ETA를 먼저 보여준다.
- **단계 완료 시 ASCII 박스 시그니처.** 중요한 git 단계(브랜치 생성/삭제·머지·배포 완료)는 닫힌 ASCII 박스(`╔═╗║╚╝`)로 한눈에 보이게.
  - 내용은 **ASCII만** — 한글·이모지·`→`(더블폭) 금지. 화살표는 `->`. 모든 줄을 같은 폭으로 패딩.
- **스킬 로드 시 스킬 전문을 응답에 출력하지 않는다.**
- **한국어로 응답한다.** 기술 용어·코드 식별자는 원형 유지.
- **불확실한 사실은 메모리/추측이 아니라 코드·문서로 검증**한 뒤 말한다.
