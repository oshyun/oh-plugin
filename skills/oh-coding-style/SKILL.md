# Coding Style

## A. 작업 전 사고방식

코드를 쓰기 전에:

- **가정하지 말 것.** 가정을 명시하고, 불확실하면 묻는다.
- **해석이 여러 갈래면 펼쳐 보인다.** 조용히 하나를 고르지 않는다.
- **더 단순한 길이 있으면 말한다.** 정당하면 푸시백한다.
- **헷갈리면 멈춘다.** 무엇이 불명확한지 짚고 묻는다.
- **수정 범위가 크면 먼저 제안한다.** 조용히 어느 방향을 택하지 않는다.
  - 외과적 변경으로 우선 처리할 수 있는지 검토한다.
  - 구조 리팩토링이 필요하다면 이유(예: 중복 누적, 의존성 역전, 확장 불가 구조 등)를 명시한 뒤 두 선택지를 사용자에게 제안한다.

판단 기준: "시니어 엔지니어가 이거 과하다고 할까?" → 그렇다면 단순화.

성공 기준을 **검증 가능한 목표**로 바꾼 뒤 그 기준이 충족될 때까지 루프한다.

- "검증 추가" → "잘못된 입력 테스트를 쓰고 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고 통과시킨다"
- "X 리팩토링" → "전후로 테스트가 통과하는지 보장한다"

## B. 코딩 패턴 / 코드 품질 규칙

### 1. 단순함 우선, 외과적 변경
- 요청된 것만. 투기적 기능·추상화·유연성·불가능 시나리오 방어 금지.
- 200줄을 50줄로 줄일 수 있으면 다시 쓴다.
- **기존 구현 요소를 먼저 살핀다.** 유사한 패턴·유틸·컴포넌트가 이미 있으면 새로 만들지 않고 재활용한다.
- **전체 아키텍처에 미치는 영향을 먼저 파악한다.** 부분 변경이 전체 구조의 일관성을 깨지 않는지 확인한다.
- 기존 코드 편집 시: 인접 코드/주석/포맷을 "개선"하지 않는다.
- 안 망가진 걸 리팩토링하지 않는다. 기존 스타일에 맞춘다.
- 내 변경이 만든 orphan(미사용 import/변수/함수)만 정리. 기존 dead code는 건드리지 않고 **언급만** 한다.
- 테스트: 바뀐 모든 줄이 요청에 직접 추적돼야 한다.

### 2. 불필요한 마이그레이션 코드 금지
- 데이터 파일을 직접 교체할 수 있으면 직접 바꾼다.
- 로더에 normalize/lazy-migration 분기를 끼워넣지 않는다 (한 번 쓰이고 영원히 남는 레거시가 됨).
- 저장 쓰기가 막히면 마이그레이션을 짜는 대신 **권한을 먼저 요청**한다. 코드는 새 스키마만 알게 유지.

### 3. 단일 출처(SSOT)
- 같은 속성의 사본을 여러 곳에 저장하지 않는다.
- 한 곳에서 조회하는 getter를 두고 하위는 그 출처를 신뢰한다.
- 규칙·설정도 단일 출처에 codify하고, 다른 문서는 *왜/사례* 보존용으로만 둔다.

### 4. UI 사전 검증 필수
- 서버에 validation이 있어도 UI에 같은 사전 검증을 넣어 입력 즉시 피드백한다 (에러 메시지 + submit 비활성).
- 빌드 없는 환경이면 로직을 복제하되 양쪽 파일 상단에 `MUST STAY IN SYNC` 주석.
- 이미 로드된 클라 state로 중복·존재 체크를 수행해 서버 왕복을 없앤다.
- 서버 에러 코드는 사용자 언어 메시지로 매핑해 우회·레이스 fallback.

### 5. 헬퍼는 객체 단일 인자, 사소한 래퍼 금지
- `a || b || fallback` 같은 사소한 표현식을 named 함수로 감싸지 않는다 (단일 사용 추상화 금지).
- 헬퍼가 정당하면(여러 호출처가 같은 객체를 다룸) 위치 인자 대신 **도메인 객체 하나**를 받아 내부에서 추출한다.
  - 예: `pickReplyText(result)` (O) vs `pickReplyText(a, b, c)` (X).

### 6. 약어·줄임말 금지, 즉시 이해 가능한 이름
- 새 식별자(변수·함수)와 주석에 의미 불명확한 약어 금지. **풀어쓴 이름** 사용 (`fb` → `fileBrowser`).
- 기존 약어를 일괄 개명할 필요는 없으나, 새로 쓰는 코드·주석은 약어 없이.
- 이름만 읽고도 **무엇을 담는지 즉시 알 수 있어야** 한다.
  - 좋은 예: `userProfileImageUrl`, `retryDelayMilliseconds`, `isSubmitButtonDisabled`
  - 나쁜 예: `data`, `value`, `flag`, `temp`, `info`, `result` (맥락 없이 범용적인 이름 금지)
- 타입·컨텍스트로 이미 명확한 부분은 반복하지 않되, 불명확하면 길어져도 풀어 쓴다.

### 7. 레거시 식별자 잔재 금지
- 기능 개편 시 옛 설계의 슬롯 번호·모드명이 이름에 남으면(예: `custom1`) 새 설계 기준으로 함께 개명한다.
- 사용자 저장값(localStorage 키 등)이 걸려 있어도 중요도 낮으면 마이그레이션 없이 키를 바꾼다.
- **키 변경으로 기존 코드가 깨진다면** 마이그레이션 스크립트 또는 서지컬 호환 코드를 추가한다.
  - 해당 코드에는 반드시 `COMPAT` 태그를 달고, 아래 정보를 모두 포함한다:
    1. 어떤 커밋(커밋 SHA 또는 PR)에서 구 식별자가 도입됐는지
    2. 마이그레이션을 추가하는 시점의 버전(앱 버전, 스키마 버전 등)과 작업 내용
    3. 언제 이 코드를 제거해도 되는지(`COMPAT-REMOVE-WHEN`)
  - 예:
    ```
    // COMPAT(custom1-slot-rename): custom1 -> primarySlot 개명 (commit abc1234, v2.3.0)
    //   구버전 저장값을 읽어 새 키로 이전 후 구 키를 삭제한다
    // COMPAT-REMOVE-WHEN: v2.3.0 이전 버전 사용률 0% 확인 후 30일 경과 시
    ```

### 8. 확장 포인트엔 관리 주석
- 화이트리스트·레지스트리 등 확장 목록의 정의부 주석에 (1) 추가 절차, (2) 플래그/데이터 흐름, (3) 엣지 케이스를 남긴다.

### 9. 부수 기능에 강한 의존성 금지
- 표시·부가 기능 때문에 런타임 의존성(예: 컨테이너 `.git` 마운트, 런타임 git 호출)을 추가하지 않는다.
- 호스트 스크립트·CI 스텝·기존 데이터 경로 재활용을 우선한다.
- "고작 이 효용에 비해 무거운 수단 아닌가"를 자문한다.

### 10. 중요도 대비 복잡도로 기각 가능
- 리뷰·simplify 패스(lint 실행 + 불필요 코드 제거)에서 발견된 사항이 추가하는 복잡도가 기능 중요도보다 크면 **한 줄 사유와 함께 기각**한다.
- 오타급·한두 줄 수정은 그대로 적용한다.

## C. 코드 주석 태그 시스템

"나중에 처리해야 할 코드"를 추적하는 태그 시스템. 흩어진 대상을 grep 한 줄로 인덱싱한다.

**공통 포맷** — 이유 줄 + 제거 조건 줄:
```
// <TAG>(<subject>): <이 코드가 존재하는 이유>
// <TAG>-REMOVE-WHEN: <이 조건이 충족되면 코드와 태그를 함께 제거>
```

전체 목록: `grep -rn "FIXME\|REFAC\|COMPAT\|QUIRK" src/`

---

**FIXME** — 알려진 버그·문제. `<subject>`에 지금 고치지 못하는 이유를 적는다.
```
// FIXME(time-constraint): 입력 범위 미검증 — 서버에서 걸러지나 클라 피드백 없음
// FIXME-REMOVE-WHEN: 클라이언트 validation 추가 시
```

**REFAC** — 기술 부채·리팩토링 대상. 왜 지금이 아닌지 이유를 포함한다.
```
// REFAC(scope-too-large): 이 함수 200줄, 3개 책임 혼재
// REFAC-REMOVE-WHEN: 해당 모듈 분리 작업 착수 시
```

**COMPAT** — 하위 호환을 위해 존재하는 코드. 구버전 클라이언트·API·스키마·설정 포맷을
지원하기 위한 분기·변환·fallback에 단다. `<subject>`에 무엇을 지원하는지 명시하고,
REMOVE-WHEN에 "언제 그 지원이 불필요해지는가"를 구체적으로 적는다.
일반 기능 코드에 달지 않는다 — "이 코드가 없으면 구버전이 깨진다"는 경우에만.
```
// COMPAT(v1-session-format): 구버전 .claude 디렉토리 구조를 신규 경로로 마이그레이션
// COMPAT-REMOVE-WHEN: v1 세션 파일을 생성하는 클라이언트 버전의 사용률이 0%가 되고
//   최소 30일 이상 경과한 뒤, 마이그레이션 로그에서 실행 흔적이 없음을 확인했을 때
```

**QUIRK** — 외부 컴포넌트(라이브러리·엔진·모델 등)의 비표준 동작을 보정·우회하는 코드.
해당 컴포넌트가 표준대로 동작했다면 없었을 코드에만 단다.
`<subject>`에 무엇이 잘못 동작하는지 적는다. 근본 원인이 우리 쪽(설정·호출 방식)이라면
출처를 함께 고치고 QUIRK를 달지 않는다.
```
// QUIRK(some-lib@2.x): 응답 Content-Type 헤더 누락 — 수동으로 주입
// QUIRK-REMOVE-WHEN: some-lib 3.0 업그레이드 후 (패치노트 #1234 확인)
```
